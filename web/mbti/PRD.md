## 1. 프로젝트 소개 및 목적

### 1.1 서비스 개요

| 항목 | 내용 |
|------|------|
| **서비스명** | MBTI 타입 발견 (가칭: TypeFinder) |
| **태그라인** | "12개의 질문으로 나를 발견하다" |
| **서비스 URL** | 미정 (예: typefinder.kr) |
| **서비스 유형** | 심리 유형 자가진단 웹 서비스 |
| **플랫폼** | 웹 (모바일 우선 반응형) |

### 1.2 서비스 비전 및 미션

**비전**: 누구나 5분 안에 자신의 성격 유형을 직관적으로 파악하고 주변과 공유할 수 있는 가장 쉬운 MBTI 플랫폼

**미션**:
- 복잡한 심리 이론을 일상적인 언어로 단순화하여 접근성을 높인다
- 결과 공유를 통해 사람들이 서로를 이해하는 대화의 시작점을 만든다
- 재미와 인사이트를 동시에 제공하는 자기 탐색 경험을 설계한다

### 1.3 핵심 가치 제안 (Value Proposition)

```
"바쁜 현대인을 위해, 12개의 핵심 질문만으로 정확도 높은 MBTI 유형을 즉시 알려주고,
감각적인 결과 카드로 SNS 공유까지 원스톱으로 제공한다."
```

**3가지 핵심 차별점**:

1. **빠름**: 12문항 / 평균 3-5분 완료 (경쟁사 대비 60% 단축)
2. **감각적인 결과**: 16가지 유형별 커스텀 비주얼 결과 카드 + 이미지 다운로드
3. **사회적 맥락**: 실시간 유형 분포 통계로 "나는 전체 중 몇 %인가?" 확인 가능

### 1.4 비즈니스 모델 개요

**1단계 (MVP)**: 무료 서비스로 트래픽 확보 및 브랜드 인지도 구축

**수익화 방향 (향후)**:
- 광고 수익 (구글 애드센스, 네이티브 광고)
- 프리미엄 상세 리포트 (심층 분석, PDF 다운로드)
- B2B 기업 채용/팀빌딩용 버전 라이선스

### 1.5 프로젝트 배경 및 문제 정의

**시장 배경**:
- MBTI는 2020년대 한국 MZ세대의 대표적 소통 코드로 자리 잡음
- "MBTI 뭐예요?"는 첫 만남의 일상적 대화 소재
- 기존 서비스 문제점:
  - 16personalities: 영어 기반, 질문 수가 60개 이상으로 너무 많음
  - 국내 서비스: UI/UX가 구식, 결과 공유 경험 미흡
  - 대부분의 서비스: 모바일 최적화 부족

**해결하는 문제**:

| 문제 | 현황 | 우리의 해결책 |
|------|------|-------------|
| 테스트가 너무 길다 | 60-100문항 | 12문항 핵심 압축 |
| 결과 공유가 불편하다 | 링크 복사에 그침 | 이미지 저장 + Web Share API |
| 디자인이 매력 없다 | 텍스트 중심 | 유형별 감각적 비주얼 카드 |
| 내가 얼마나 희귀한지 모른다 | 통계 없음 | 실시간 유형 분포 통계 |

### 1.6 성공 지표 (KPI)

**런칭 3개월 목표**:

| 지표 | 목표값 | 측정 방법 |
|------|--------|---------|
| 월간 활성 사용자 (MAU) | 10,000명 | Google Analytics |
| 테스트 완료율 | 75% 이상 | 시작 대비 결과 페이지 도달률 |
| 결과 공유율 | 30% 이상 | 공유 버튼 클릭 / 결과 조회 |
| 평균 세션 시간 | 3분 이상 | GA 세션 데이터 |
| 재방문율 | 20% 이상 | 30일 내 재방문 |

## 2. 대상 사용자

### 2.1 타겟 사용자 페르소나

#### 페르소나 1: 이민지 (20대 대학생)

```
나이: 22세 | 직업: 대학교 3학년
생활 패턴: 스마트폰 중심 생활, SNS 일일 2-3시간
기술 친숙도: 높음 (디지털 네이티브)
주요 기기: iPhone + MacBook
```

**목표(Goals)**:
- 새로 사귄 친구, 선배와의 관계에서 자신을 빠르게 소개하고 싶다
- MBTI 결과를 인스타그램 스토리에 올려 소통하고 싶다
- 자신의 성격 강점을 확인하고 진로 탐색에 활용하고 싶다

**불편함(Pain Points)**:
- 기존 테스트가 너무 길어서 중간에 포기한 적이 있다
- 결과 이미지가 촌스러워서 SNS에 올리기 창피하다
- 친구와 결과를 비교하기 어렵다

**핵심 사용 시나리오**: 쉬는 시간 5분, 스마트폰으로 테스트 후 인스타 스토리 공유

**핵심 사용 시나리오**: 팀 회의 전 5분 워밍업으로 전체 팀원에게 링크 공유

### 3.2 기능별 상세 설명

#### F-001: 테스트 플로우

**질문 구성 (12문항)**:

각 축별 3개 질문, 총 12문항 구성:

| 축 | 질문 수 | 판별 내용 |
|----|---------|---------|
| E (외향) / I (내향) | 3문항 | 에너지 방향, 사교 성향 |
| S (감각) / N (직관) | 3문항 | 정보 수집 방식, 현실 vs 가능성 |
| T (사고) / F (감정) | 3문항 | 의사결정 방식, 논리 vs 감정 |
| J (판단) / P (인식) | 3문항 | 생활 방식, 계획 vs 즉흥 |

**질문 샘플**:

```
Q1 (E/I): 주말에 가장 즐거운 시간은?
  A. 친구들과 왁자지껄한 파티 (E)
  B. 혼자만의 조용한 취미 시간 (I)

Q4 (S/N): 새 가전제품을 살 때 나는?
  A. 스펙표를 꼼꼼히 비교하고 결정 (S)
  B. 직감과 브랜드 이미지로 결정 (N)
```

**점수 계산 방식**:
- 각 축 3문항, 문항당 +1점
- 예: E 3점 vs I 0점 → 강한 외향형
- 동점(1.5/1.5) 시: 마지막 질문 결과 우선

#### F-002: 프로그레스 바

- 현재 문항 / 전체 문항 수 텍스트 표시 (예: "3 / 12")
- 진행률 시각적 바 (0% → 100%)
- 각 문항 완료 시 부드러운 증가 애니메이션 (300ms ease)

#### F-003: 결과 분석 엔진

**계산 알고리즘**:
```
점수 집계:
- E_score = E 답변 수 (0-3)
- I_score = 3 - E_score
- S_score, N_score, T_score, F_score, J_score, P_score 동일 방식

유형 결정:
- dim1 = E_score >= 2 ? 'E' : 'I'  (동점=2 이상이면 E)
- dim2 = S_score >= 2 ? 'S' : 'N'
- dim3 = T_score >= 2 ? 'T' : 'F'
- dim4 = J_score >= 2 ? 'J' : 'P'
- mbtiType = dim1 + dim2 + dim3 + dim4  // 예: "ENFJ"

점수 비율 (결과 페이지 표시용):
- E 비율 = (E_score / 3) * 100  // 예: 67%
```

#### F-004: 결과 페이지

각 MBTI 유형별 콘텐츠 구성:

| 섹션 | 내용 | 분량 |
|------|------|------|
| 유형 뱃지 | 4글자 코드 + 별명 (예: "ENFJ - 선도자") | 1줄 |
| 핵심 설명 | 이 유형의 핵심 특징 요약 | 2-3문장 |
| 강점 | 3-5가지 불릿 리스트 | 3-5개 항목 |
| 약점 | 3-5가지 불릿 리스트 | 3-5개 항목 |
| 유명인 | 이 유형으로 알려진 유명인 2-3명 | 이미지 + 이름 |
| 추천 직업 | 적합한 직업군 5-8개 | 태그 형태 |
| 궁합 유형 | 잘 맞는 유형 / 주의할 유형 | 유형 코드 카드 |
| 비율 표시 | E/I, S/N, T/F, J/P 각 축 퍼센트 바 | 4개 게이지 바 |

#### F-005: 이미지 저장

- **기술**: `html2canvas` 라이브러리 활용
- **저장 영역**: 결과 카드 영역만 캡처 (전체 페이지 아님)
- **파일명**: `mbti-result-{TYPE}-{날짜}.png` (예: `mbti-result-ENFJ-20260308.png`)
- **해상도**: 2x 배율(고해상도) 렌더링
- **포함 내용**: 유형 코드, 별명, 핵심 설명, 강점 TOP3, 축 퍼센트, 서비스 로고/URL

#### F-006: 결과 공유

| 공유 방법 | 기술 | 우선순위 |
|----------|------|---------|
| 링크 복사 | Clipboard API | 최우선 |

## 4. UI/UX 컨셉 및 디자인 가이드라인

### 4.1 디자인 원칙

**1. 심플 & 포커스 (Simplicity First)**
- 한 화면에 하나의 핵심 행동만 제시
- 불필요한 요소 제거, 핵심에 집중

- 16가지 유형별 고유한 색상과 비주얼 정체성
- 마이크로 애니메이션으로 생동감 부여

**3. 모바일 퍼스트 (Mobile First)**
- 엄지 손가락 하나로 조작 가능한 UI
- 터치 타겟 최소 44x44px

**4. 즉각적 피드백 (Immediate Feedback)**
- 답변 선택 즉시 시각적 반응
- 진행 상황 상시 노출

**5. 공유 최적화 (Share-Optimized)**
- 결과 화면이 곧 공유할 이미지
- OG 이미지 자동 생성

| CTA 버튼 | "제출" | "테스트 시작하기" |
| 로딩 중 | "로딩 중..." | "당신의 답변을 분석하고 있어요 ✨" |
| 이미지 저장 완료 | "저장됨" | "이미지가 저장됐어요! 인스타에 공유해보세요" |
| 오류 발생 | "오류 발생" | "일시적인 문제가 생겼어요. 잠시 후 다시 시도해주세요" |
| 링크 복사 완료 | "복사됨" | "링크를 복사했어요! 친구에게 공유해보세요" |
| 재시작 유도 | "처음부터" | "다시 테스트하기" |
| 빈 통계 | "데이터 없음" | "아직 참여자가 없어요. 첫 번째 주인공이 돼보세요!" |

**금지 표현**:
- "Error", "Failed" 등 기술적 용어 노출 금지
- 과도한 이모지 (문장당 최대 1개)
- 단정적 표현 ("당신은 ~입니다" → "~한 경향이 있어요")
- MBTI를 절대적 진실로 표현하는 문구

## 5. 기술 스택 제안

### 5.1 프론트엔드

| 기술 | 선택 | 근거 |
|------|------|------|
| **프레임워크** | Next.js 15 (App Router) | SSR로 OG 태그 동적 생성 필수, SEO 최적화, Vercel 배포 최적화 |
| **언어** | TypeScript 5 | 타입 안정성, 16개 유형 상수 타입 관리 |
| **스타일링** | Tailwind CSS v4 | 유틸리티 클래스, 빠른 UI 구현, 번들 최소화 |
| **UI 컴포넌트** | shadcn/ui | 접근성 내장, 커스터마이즈 용이 |
| **상태 관리** | Zustand | 경량, 테스트 진행 상태 관리에 충분 |
| **애니메이션** | Framer Motion | 페이지 전환, 카드 인터랙션 |
| **이미지 캡처** | html2canvas | 결과 카드 PNG 저장 |
| **차트** | Recharts | 통계 페이지 유형 분포 차트 |

**선택 근거 상세**:
- **Next.js 선택 이유**: 결과 URL(`/result/ENFJ`)마다 고유한 OG 메타태그를 서버에서 생성해야 카카오/인스타 공유 시 미리보기가 정확히 노출됨. Pure React(CSR)로는 이 기능 구현 불가.
- **Tailwind 선택 이유**: 16가지 유형별 동적 색상 변경을 인라인 스타일 없이 구현 가능 (CSS 변수 + Tailwind JIT).

### 5.2 백엔드

| 기술 | 선택 | 근거 |
|------|------|------|
| **API** | Next.js API Routes | 별도 백엔드 불필요, 동일 레포 관리 |
| **데이터베이스** | Supabase (PostgreSQL) | 무료 플랜으로 MVP 시작, 실시간 기능, 관리 콘솔 편리 |
| **ORM** | Prisma | 타입 안전한 DB 쿼리, 마이그레이션 관리 |

**대안 검토**:
- Firebase Firestore: NoSQL이라 통계 집계 쿼리에 불리
- PlanetScale: MySQL 기반, 다만 무료 플랜 종료됨 (2024)
- **결론**: Supabase가 PostgreSQL 집계 함수 + 무료 플랜 + 관리 UI 측면에서 MVP에 최적

## 7. 데이터 모델

### 7.1 핵심 데이터 엔티티

서비스의 핵심 저장 데이터는 **최소화 원칙**을 따릅니다. 개인 식별 정보는 일절 수집하지 않습니다.

#### 엔티티 1: TestResult (테스트 결과)

통계 집계를 위한 익명 결과 저장. 개인을 식별할 수 없는 정보만 포함.

```sql
CREATE TABLE test_results (
  id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  mbti_type   CHAR(4)     NOT NULL,  -- 예: 'ENFJ'
  e_score     SMALLINT    NOT NULL CHECK (e_score BETWEEN 0 AND 3),
  s_score     SMALLINT    NOT NULL CHECK (s_score BETWEEN 0 AND 3),
  t_score     SMALLINT    NOT NULL CHECK (t_score BETWEEN 0 AND 3),
  j_score     SMALLINT    NOT NULL CHECK (j_score BETWEEN 0 AND 3),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
  -- IP, 기기 정보 등 개인 식별 데이터 저장 안 함
);
```

#### 엔티티 2: MbtiType (MBTI 유형 메타데이터)

16개 유형의 콘텐츠 데이터. 애플리케이션 코드 내 JSON 상수로 관리 (DB 불필요).

```typescript
// types/mbti.ts
interface MbtiTypeData {
  code: string;          // 예: 'ENFJ'
  nickname: string;      // 예: '선도자'
  tagline: string;       // 예: '카리스마와 공감 능력을 겸비한 천생 리더'
  description: string;   // 2-3문장 설명
  strengths: string[];   // 3-5가지 강점
  weaknesses: string[];  // 3-5가지 약점
  careers: string[];     // 추천 직업 5-8개
  compatibleTypes: string[];   // 잘 맞는 유형 코드 배열
  challengingTypes: string[];  // 주의할 유형 코드 배열
  color: {
    primary: string;     // HEX 색상
    secondary: string;
    gradient: string;    // CSS gradient 값
  };
  celebrities: Celebrity[];    // 유명인 2-3명
}

interface Celebrity {
  name: string;
  role: string;   // 예: '영화감독', '가수'
}
```

#### 엔티티 3: Question (질문 데이터)

애플리케이션 코드 내 JSON 상수로 관리.

```typescript
// data/questions.ts
interface Question {
  id: number;           // 1-12
  text: string;         // 질문 텍스트
  axis: 'EI' | 'SN' | 'TF' | 'JP';  // 해당 축
  optionA: {
    text: string;
    icon: string;       // 이모지 또는 아이콘 이름
    dimension: 'E' | 'S' | 'T' | 'J';  // 해당 차원
  };
  optionB: {
    text: string;
    icon: string;
    dimension: 'I' | 'N' | 'F' | 'P';
  };
}
```

### 7.2 데이터 흐름

```
[클라이언트]
테스트 진행 중:
  - answers: Record<number, 'A'|'B'>  → Zustand 상태 관리
  - 브라우저 이탈 시 → localStorage 자동 저장

테스트 완료 시:
  1. 클라이언트에서 유형 계산 (순수 함수)
  2. POST /api/results → TestResult DB 저장 (익명)
  3. /result/{TYPE} 페이지로 라우팅

[서버 - API Routes]
GET  /api/stats       → TestResult 집계 조회 (1시간 캐시)
POST /api/results     → 새 결과 저장 (Rate Limit 적용)
```

### 7.5 개인정보 처리 방침

- **개인 식별 정보 미수집**: IP 주소, 기기 정보, 사용자 계정 없음
- **수집 데이터**: MBTI 유형 코드 + 각 축 점수 + 생성 시각 (익명)
- **법적 검토**: 개인정보보호법상 개인정보에 해당하지 않으나, 서비스 하단에 "이 서비스는 개인 식별 정보를 수집하지 않습니다" 명시 권장
- **고지 문구**: "본 테스트는 엔터테인먼트 목적이며, 전문 심리 진단을 대체하지 않습니다"
